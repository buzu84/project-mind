# Backend Glossary

Terms explained for developers coming from a frontend background.

---

### Route Handler
A server-side function in Next.js that handles HTTP requests. Lives in `src/app/api/.../route.ts`. Equivalent to an Express.js route handler but colocated with the frontend. Exports named functions: `GET`, `POST`, `PATCH`, `DELETE`.

### RLS (Row-Level Security)
A PostgreSQL feature that adds access control rules directly to database tables. Each table has policies like "users can only see rows where `user_id` matches their auth ID." Even if application code has a bug, the database enforces isolation.

### pgvector
A PostgreSQL extension that adds vector data types and similarity search. Stores 1536-dimensional embedding vectors alongside regular data. Enables "find chunks similar to this query" without a separate vector database.

### Embedding
A numeric vector (array of 1536 floats) that represents the semantic meaning of text. Two texts about similar topics will have vectors that are geometrically close (high cosine similarity). Generated by OpenAI's `text-embedding-3-small` model.

### RAG (Retrieval-Augmented Generation)
A pattern where you first **retrieve** relevant documents from a database, then **augment** the AI prompt with those documents, then **generate** a response. This grounds the AI in actual data instead of generic knowledge.

### Orchestration
Coordinating multiple steps in sequence to complete a complex task. The Decision Review orchestrates: auth → load data → retrieve evidence → call AI → validate → retry → save → cleanup → track usage. Each step depends on the previous one.

### Normalization
Transforming messy data into a consistent format. AI outputs often have inconsistencies (snake_case vs camelCase, "65" vs 65, "legal" vs "business"). The normalization layer fixes these before validation.

### Schema Validation (Zod)
Using the Zod library to verify that data matches an expected shape. If the AI returns `confidenceScore: "high"` instead of `confidenceScore: 75`, Zod catches it. Applied to both incoming API requests and AI outputs.

### Structured Output
Requesting that an LLM return data in a specific JSON format rather than free-form text. OpenAI's `response_format: { type: "json_object" }` forces JSON, but doesn't guarantee the schema — that's why Zod validation + normalization is needed.

### Retry Logic
If the AI returns invalid output, the system sends a correction prompt explaining what was wrong and asks for a fixed response. Maximum 1 retry. The correction prompt includes the specific Zod validation errors.

### Middleware
Code that runs before route handlers. In ProductMind, Edge Middleware refreshes the Supabase auth session cookie on every request, preventing session expiration during active use.

### Persistence
Saving data to the database. In Decision Review, this uses an **insert-before-delete** strategy: new records are inserted first, then old ones are deleted. If the insert fails, the old data remains intact.

### Insert-Before-Delete (Atomic-ish Save)
A data integrity pattern. Instead of deleting old records then inserting new ones (which risks data loss if the insert fails), we insert first, verify success, then clean up old records. Not truly atomic (no transaction), but practically safe for the use case.

### Serverless Cold Start
When a serverless function hasn't been called recently, the runtime shuts it down. The next request has a "cold start" delay while the runtime initializes. Relevant because in-memory state (like rate limit counters) is lost on cold start.

### Rate Limiting
Restricting how many requests a user can make in a time window. ProductMind uses a sliding window algorithm with two tiers: standard (20/hr) and heavy (5/15min). Prevents abuse and controls AI API costs.

### Sliding Window
A rate limiting algorithm that tracks requests in a rolling time window. Unlike fixed windows (which reset at clock boundaries), sliding windows prevent burst-at-boundary attacks. Implemented as an in-memory map of timestamps per user.

### Lexical Guard
A secondary quality check in the RAG pipeline. After semantic similarity search returns candidate chunks, the lexical guard verifies they share actual keywords with the query. Catches false-positive semantic matches where vector similarity is high but the content is topically irrelevant.

### Quality Gate
A minimum threshold for accepting retrieved evidence. Chunks below `MIN_PROMPT_SIMILARITY` (0.2 cosine similarity) are discarded. Prevents injecting irrelevant context that could mislead the AI.

### Citation Sanitization
After the AI generates a response, citation references (like `[1]`, `[2]`) are checked against the actual evidence provided. Hallucinated citations (referencing evidence that wasn't provided) are stripped out.

### `generated_by` Marker
A column on AI-generated database records that identifies their origin (e.g., `"decision_review_v1"`). Used during cleanup to distinguish AI-generated records from user-created ones, ensuring re-analysis doesn't delete user data.

### Defense-in-Depth
Using multiple layers of security so that if one fails, others still protect the system. In ProductMind: RLS policies + explicit `user_id` query filters + auth middleware. Any single layer failing doesn't expose data.

### Mock AI
A development mode (`USE_REAL_AI=false`) that returns deterministic fake responses instead of calling OpenAI. Enables development without API costs, predictable testing, and offline work.
