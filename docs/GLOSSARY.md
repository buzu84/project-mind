# Glossary

Practical definitions for terms used in the ProductMind codebase. Aimed at engineers working on the project, not textbook definitions.

---

## Runtime & Framework

### Route Handler
A server-side function in `src/app/api/.../route.ts` that handles HTTP requests. Exports named functions (`GET`, `POST`, `PATCH`, `DELETE`). Functionally equivalent to Express route handlers but colocated with the frontend in a single Next.js deployment.

### Server Component
A React component that renders on the server (the default in Next.js App Router). Can directly `await` database queries, call `getCurrentUser()`, and read cookies. Cannot use `useState`, `useEffect`, or browser APIs. Most pages in `src/app/(dashboard)/` are server components.

### Client Component
A React component marked with `"use client"` that runs in the browser. Required for interactivity: forms, buttons, SSE stream consumption, `useState`/`useEffect`. AI tool forms and the chat interface are client components. They call API routes via `fetch()`.

### Server Action
A server-side function marked with `"use server"` that can be called directly from React forms without writing an API route. Used for simple CRUD: project create/edit/delete, feedback documents, feature ideas, and project context.

### Edge Middleware
Code in `src/middleware.ts` that runs before every request (on Vercel's edge network). Refreshes the Supabase auth session cookie, redirects unauthenticated users from protected paths to `/sign-in`, and redirects authenticated users away from auth pages to `/dashboard`.

### Service Layer (`src/lib/`)
Reusable backend logic separated from HTTP/form concerns. Route handlers and Server Actions validate input, then delegate to service layer files. Key modules: `lib/decisions/` (Decision Engine), `lib/rag/` (RAG pipeline), `lib/evidence/` (evidence retrieval), `lib/ai/` (rate limiting, usage tracking).

## AI & Retrieval

### RAG (Retrieval-Augmented Generation)
The pattern of **retrieving** relevant documents from the database, then **augmenting** the AI prompt with them, before **generating** a response. In ProductMind: feedback documents → chunked → embedded → stored in pgvector → retrieved by similarity → injected into prompt. Implemented in `src/lib/rag/`. See [RAG_ARCHITECTURE.md](architecture/RAG_ARCHITECTURE.md) for full pipeline details.

### Embedding
A 1536-dimensional float vector representing the semantic meaning of a text chunk. Generated by OpenAI's `text-embedding-3-small` model. Two texts about similar topics produce geometrically close vectors (high cosine similarity). Stored in `document_chunks.embedding`.

### pgvector
A PostgreSQL extension that adds vector column types and similarity search operators. Supabase includes it by default. Used via the `match_document_chunks` RPC function which performs cosine similarity search filtered by `project_id`.

### Evidence Layer
An abstraction in `src/lib/evidence/` that wraps RAG with intent-specific config. Each caller (chat, decision review, roadmap) can request evidence with different retrieval parameters (result count, similarity threshold) via a `RetrievalIntent`. See [RAG_ARCHITECTURE.md](architecture/RAG_ARCHITECTURE.md).

### Quality Gate
The minimum cosine similarity threshold (`MIN_PROMPT_SIMILARITY = 0.2`) for accepting RAG results. Chunks below this are discarded to prevent injecting irrelevant context. If no chunks pass, the AI proceeds without evidence.

### Lexical Guard
A secondary check after vector search. Verifies that retrieved chunks share actual keywords with the query, catching false-positive semantic matches. Implemented in `src/lib/rag/context-builder.ts`.

### Normalization Layer
Code in `src/lib/decisions/review-normalize.ts` that transforms messy AI output into a consistent format before Zod validation. Handles: `snake_case` → `camelCase` key mapping, string-to-number conversion (`"65"` → `65`), enum alias mapping (`"legal"` → `"business"`), and null-to-empty-array defaults. ~30 aliases mapped to 8 valid assumption types.

### Structured AI Response Validation
The pipeline that validates AI-generated JSON: `JSON.parse()` → normalization → `Zod.safeParse()` → retry on failure. Applied to Decision Review, Insights, Roadmap, Multi-Agent Review, and Feature Scoring. Routes that return markdown (PRD, Competitive Analysis) skip this.

### Citation Sanitization
After AI generates a response referencing evidence as `[1]`, `[2]`, etc., these citation IDs are checked against the evidence actually provided. Hallucinated citations are stripped. Implemented in `src/lib/evidence/citations.ts`. Note: citations are used in AI prompts only — not rendered in the UI.

### SSE (Server-Sent Events)
A streaming protocol where the server pushes chunks to the client over a long-lived HTTP connection. Used by `/api/ai/chat` and `/api/ai/global-chat` to stream AI responses token-by-token. Each chunk is `data: "text"\n\n`. The final event is `data: {"done":true,"id":"msg-uuid"}\n\n`.

## Data & Security

### RLS (Row-Level Security)
PostgreSQL policies that control which rows a user can access. Every table has policies like `auth.uid() = user_id`. The Supabase client automatically passes the user's JWT. Even buggy application code can't leak other users' data.

### Project-Scoped Query
A database query that filters by `project_id` to ensure data stays within one project. All AI tools, feedback, features, and decisions are project-scoped. Pattern: `.eq("project_id", projectId).eq("user_id", user.id)`.

### Defense-in-Depth
Three layers of access control: (1) RLS policies on every table, (2) explicit `.eq("user_id", user.id)` query filters, (3) `getCurrentUser()` auth check in every route. Any single layer failing doesn't expose data.

### Supabase Server Client
Created by `createClient()` in `src/lib/supabase/server.ts`. Reads the session cookie and passes it to Supabase so queries respect RLS. Also provides a service-role (admin) client that bypasses RLS — used only for account deletion.

### Supabase Browser Client
Created by `createClient()` in `src/lib/supabase/client.ts`. Used in client components for auth operations (sign-in, sign-up, sign-out, password reset).

## Decision Engine

### `decisions` Table (Legacy)
A catch-all table storing generated AI artifacts (PRDs, competitive analyses, prioritization outputs). Named "decisions" from an early version. Each row has `type: "PRD" | "COMPETITIVE_ANALYSIS" | "PRIORITIZATION"`, `input`, and `output` JSON columns. **Not related to** `product_decisions`.

### `product_decisions` / Decision Engine
A separate, normalized data model for structured product decisions. Seven tables: `product_decisions`, `product_decision_options`, `product_assumptions`, `product_evidence`, `product_decision_evidence_links`, `product_decision_recommendations`, `product_decision_agent_reviews`. See [DECISION_REVIEW_FLOW.md](architecture/DECISION_REVIEW_FLOW.md).

### Insert-Before-Delete
The save strategy for Decision Review re-analysis. New records are inserted first; only after success are old records deleted. Prevents data loss if the insert fails. Not a true database transaction — Supabase JS client doesn't support multi-statement transactions.

### `generated_by` Marker
A column on AI-generated records (value: `"decision_review_v1"`). Cleanup only deletes rows with this marker (or `NULL` for legacy). Ensures future user-created records won't be accidentally deleted during re-analysis.

## Operations

### Rate Limiting Tiers
Two tiers in `src/lib/ai/rate-limiter.ts`:
- **Standard** (20/hour): Chat, Global Chat, Insights, Feature Scoring, Prioritization
- **Heavy** (5/15min): PRD, Roadmap, Competitive Analysis, Multi-Agent Review, Decision Review

Admin bypass via `ADMIN_EMAILS` env var. In-memory sliding window — resets on serverless cold start. See [RATE_LIMITING.md](api/RATE_LIMITING.md).

### Mock AI Mode
When `USE_REAL_AI=false` (development only), AI routes return deterministic fake responses from `src/lib/ai/mock-*.ts` files. Tracked in `ai_usage` with `is_mock: true`. Production always uses real AI — `isRealAI()` returns `true` when `NODE_ENV=production`.

### AI Usage Tracking
Every AI call (real or mock) records telemetry to the `ai_usage` table: model, tokens, cost, latency, feature name, status, mock flag. Tracked via `void trackAIUsage(...)` (fire-and-forget). Visible on `/usage` page and dashboard activity feed.

### Fire-and-Forget
The pattern of calling an async function with `void` prefix: `void trackAIUsage(...)`. The function runs in the background — if it fails, the main request is unaffected. Used for usage tracking so telemetry never blocks responses.

### Serverless Cold Start
When a Vercel function hasn't been invoked recently, its container is shut down. The next request initializes a new container. In-memory state (rate limit counters) is lost, making rate limits softer than configured.

### Zod Schema
A runtime validation schema using the Zod library. Applied to: (1) incoming API request bodies, and (2) AI-generated JSON responses before saving. If validation fails: 400 for user input, retry with correction prompt for AI output.
